# Expert Feedback And Resume Guard Design

## 背景

当前 Expert 对话链路存在三类高频问题：

1. 流式回复因为 `load failed`、网络中断或用户手动停止而留下半截内容，用户希望点击“继续”时先判断这条内容是否其实已经完整，而不是直接重新走工具链。
2. 有些消息虽然状态不是 `partial`，但从用户视角看已经明显被截断，缺少标准化的问题反馈入口，也缺少一套便于后续排查和喂给 Codex 的上下文采集机制。
3. 澄清流程存在明显交互缺陷，包括“问句没有选项”“单选点一下直接进入下一步”“带子选项的选择无法取消”等。

本设计将上述问题统一收敛到一个“问题反馈 + 续写守门 + 澄清修复”闭环中。

## 目标

- 为 Expert 回复提供标准化问题反馈入口，自动附带完整上下文。
- 在“继续生成”之前先做一次“免工具完整性检查”，避免不必要的工具调用和重复分析。
- 为被用户判断为疑似截断的 `completed` 消息提供“检查并继续”的通道。
- 为管理员提供受控的反馈查看与处理入口。
- 修复澄清流程的单选确认、空选项兜底、子选项取消等交互缺陷。

## 非目标

- 不把反馈系统扩展到 debate、agent、analysis 等全部模块。
- 不引入复杂权限系统，仅支持固定管理员账号访问反馈后台。
- 不对现有 Expert 工具链做大范围重构，仅在恢复链路上增加“完整性检查”守门。

## 方案概览

方案由三部分组成：

1. `expert.feedback_reports` 反馈存储与 `Admin` 管理入口。
2. `resume` 前置完整性检查与无工具续写守门。
3. 澄清交互状态机修复与前后端兜底。

## 数据与权限设计

### 默认管理员

- 启动时自动确保存在固定账号：
  - 用户名：`Admin`
  - 密码：`WeigongAdmin`
- 若账号已存在，则不覆盖原有密码哈希。
- 该账号通过现有 `/api/v1/app/login` 登录流程进入系统，无需新增认证方式。

### 反馈存储

反馈数据写入 `expert_chat.duckdb`，新增表 `expert.feedback_reports`。

字段设计如下：

- `id`: 反馈 ID
- `user_id`: 提交反馈的用户
- `session_id`: 会话 ID
- `message_id`: 关联的 expert 消息 ID
- `expert_type`: `rag` / `short_term` / `data` / `quant` / `info` / `industry`
- `report_source`: `reply` / `clarification` / `resume`
- `issue_type`: `load_failed` / `llm_truncated` / `resume_misjudged_complete` / `clarify_missing_options` / `clarify_auto_advance` / `clarify_subchoice_stuck` / `other`
- `user_note`: 用户补充描述
- `user_message`: 对应用户原问题
- `assistant_content`: 当前 AI 内容
- `message_status`: `partial` / `completed` / 其他状态快照
- `thinking_json`: 当前消息思考过程完整快照
- `context_json`: 结构化上下文快照
- `created_at`: 创建时间
- `resolved_at`: 处理时间
- `resolver`: 处理人
- `resolution_note`: 处理备注

### 反馈上下文规范

`context_json` 需要保存后续分析和复现最有用的原始信息，不做截断：

- 当前用户问题
- 当前 AI 内容
- 当前消息状态
- 当前消息 `thinking`
- 最近几条对话历史
- 当前专家类型
- 前端错误名与错误文本
- 是否执行过继续生成
- 当前澄清卡片原始数据
- 当前澄清选择结果

### 权限控制

- 新增后台反馈列表与详情接口，仅允许 `Admin` 访问。
- 非 `Admin` 访问反馈管理接口一律返回 `403`。
- 普通用户仅能提交反馈，不能浏览其他反馈记录。

## 续写守门设计

### 问题

现有 `/api/v1/expert/chat/resume` 直接续写 `partial` 消息，没有先判断当前内容是否已经足够完整。
用户遇到 `load failed` 或模型自然截断时，也缺少对 `completed` 消息触发检查的路径。

### 新流程

把现有 `resume` 升级为“两阶段恢复”：

1. **完整性检查**
   - 输入：用户原问题、最近对话历史、当前 AI 内容、当前消息状态、思考摘要。
   - 约束：绝不调用工具、不跑 `recall_and_think`、不重新分析股票。
   - 输出：结构化 JSON，例如：
     - `is_complete`
     - `reason`
     - `confidence`
     - `resume_from_hint`

2. **条件续写**
   - 如果 `is_complete=true`：
     - 直接将消息状态更新为 `completed`
     - 返回完成事件，不进入续写
   - 如果 `is_complete=false`：
     - 走现有 `resume_reply` 续写
     - 续写仍然只基于已有上下文，不走工具链

### 适用范围

- `partial` 消息点击“继续生成”
- 用户对 `completed` 消息判断为疑似截断时触发“检查并继续”
- 因 `load failed`、前端 SSE 中断等造成的半成品消息恢复

## 反馈交互设计

### 消息级反馈入口

每条 expert 回复下方新增红色三角感叹号按钮，点击后弹出轻量反馈面板。

默认问题类型：

- 加载失败中断
- LLM 回复像是截断了
- 点继续后其实已经完整
- 澄清问题没有选项
- 点一个选项就自动下一步了
- 子选项选了后没法取消
- 其他

### 提交行为

点击提交时，前端自动附带上下文，不要求用户手工组织排查信息。

对于“加载失败中断”和“LLM 回复像是截断了”两类问题，面板提供“提交并检查续写”操作，直接走新的完整性检查流程。

## 管理后台设计

新增后台页面 `/admin/feedback`，只对 `Admin` 显示导航入口。

页面能力：

- 列表查看反馈
- 按 `issue_type`、`expert_type`、处理状态筛选
- 打开反馈详情查看完整上下文 JSON
- 标记已处理并填写备注

管理后台重点是让后续人工筛选和喂给 Codex 更高效，不追求复杂运营功能。

## 澄清交互修复设计

### 单选确认

- 单选不再“点击即提交”。
- 统一改成“先选中，再点击确认按钮提交”。
- 多选和单选都走同一套“选择态 -> 确认态”模型，减少状态分叉。

### 子选项取消

- 带 `sub_choices` 的选项，父项仅负责展开/折叠。
- 子选项被选中后，再点击一次同一子选项即可取消。
- 也可提供父级清除入口，回到未选择状态。

### 空选项兜底

如果后端生成结果为：

- `should_clarify=true`
- 但 `options` 为空

则不能直接进入下一步，需按以下顺序处理：

1. 后端优先做一次保底兜底，生成通用但可用的选项。
2. 如果兜底仍失败，前端展示错误态与反馈入口。
3. 前端不得在空选项场景下自动推进到 chat。

## 错误处理

- 反馈写入失败不影响当前聊天主流程，但需要在日志中打印完整错误。
- 完整性检查失败时，不默认重跑工具，而是把消息保持原状态并给用户明确提示。
- 续写再次中断时，继续保存最新 partial 内容。
- 管理后台接口统一记录权限拒绝与处理失败日志。

## 测试策略

### 后端 pytest

- 默认 `Admin` 自动创建
- 反馈表初始化与写入
- 非 `Admin` 访问后台接口返回 `403`
- `resume` 在“已完整”场景不触发续写
- `resume` 在“不完整”场景才触发续写
- `should_clarify=true + 空 options` 的兜底行为

### 前端 node:test

- 单选选择后不会立即提交
- 单选需要确认才继续
- 子选项可二次点击取消
- 父选项展开不代表已选中
- 反馈 payload 自动带齐 message/status/thinking/error/session 上下文
- “提交并检查续写”在已完整场景只结束消息，不追加新内容

### 集成回归

- Expert 页面：中断消息 -> 点击继续 -> 完整性检查 -> 完整则完成 / 不完整则续写
- 澄清页面：空选项不自动推进，单选确认后才继续

## 风险与控制

- 固定默认管理员账号安全性较低，但这是用户明确要求；后续若用于公网部署，应改为环境变量或首次安装向导。
- 完整性检查仍依赖 LLM 判断，可能存在误判，因此返回结果应保留 `reason` 和 `confidence` 供日志与后台复盘。
- 反馈上下文保存量较大，但必须遵守“不截断原始数据”的项目铁律。
