### 分层说明

| 层级 | 职责 | LLM 使用模式 |
|------|------|-------------|
| 用户前端 | 展示、交互 | 无 |
| 专家层 | 跨引擎综合分析、对话 | 有状态对话，需要 Memory |
| 业务引擎 | 各自领域的计算和分析 | 可选，无状态调用（分类/提取/生成） |
| 数据引擎 | 数据获取、持久化、缓存 | 无（纯存储服务） |

关键区别于旧版：
- **数据引擎下沉为纯基础设施**，不和业务引擎并列
- **每个业务引擎预留 LLM 接口**，但默认不启用，可按需插入
- **LLM 调用结果缓存在数据引擎**（通用存储服务），避免各引擎自建缓存
- **Memory 只在专家层**，业务引擎对 LLM 的调用是无状态的

## 各引擎职责

### 数据引擎 — 基础设施层

- 全市场行情拉取（Tencent/AKShare/BaoStock 三级降级）
- DuckDB 持久化（快照、日线、历史）
- 公司概况管理
- LLM 调用结果缓存（各引擎共享）
- 独立 REST API (`/api/v1/data/*`)
- **设计目标：可被第三方替换，只要接口契约一致**

### 聚类引擎 — 基本面

- BGE 语义嵌入 + HDBSCAN 聚类 + UMAP 降维
- 3D 地形插值
- 基本面特征（PE/PB/市值/换手率等）
- 多归属软隶属度
- 依赖数据引擎获取原始数据
- **LLM 接入点：** 聚类结果自然语言解读、行业标签自动生成

### 信息引擎 — 消息面（待建设）

- 新闻/公告数据源接入
- 事件提取（并购、增减持、业绩预告等）
- 行业/个股舆情热度
- 独立 REST API (`/api/v1/news/*`)
- **LLM 接入点：** 情感分析（利好/利空/中性）、新闻摘要生成、事件分类

### 量化引擎 — 技术面（待增强）

- 技术指标计算（MA/RSI/MACD/布林带等）
- 多因子模型（当前 13 因子，可扩展）
- IC/ICIR 因子回测
- 量化信号生成
- 独立 REST API (`/api/v1/quant/*`)
- **LLM 接入点：** 因子含义解读、回测结果分析报告

### 专家层 — AI 分析师

- MCP Server 提供工具访问各引擎
- LLM 综合三个面（基本面+消息面+技术面）进行分析
- **Conversation Memory：** 维护对话上下文，跨会话记忆
- 支持两种交互方式：
  - Agent 对话（Claude Code / MCP Client）
  - 前端对话面板（需 LLM API 支持）
- RAG 增强（远期）：历史分析报告、研报知识库

## LLM 接入设计

### 统一 LLM 接口

各引擎通过统一抽象调用 LLM，而非各自实现：

```python
class LLMCapability:
    """引擎可选的 LLM 能力 — 统一接口"""

    def __init__(self, cache_store=None):
        self._provider = None       # 延迟初始化，不配置则不启用
        self._cache = cache_store   # 数据引擎提供的缓存

    @property
    def enabled(self) -> bool:
        return self._provider is not None

    def complete(self, prompt: str, cache_key: str = None) -> str:
        """无状态 LLM 调用（带可选缓存）"""

    def classify(self, text: str, categories: list[str]) -> str:
        """分类任务"""

    def extract(self, text: str, schema: dict) -> dict:
        """结构化提取"""
```

### 设计原则

- **默认关闭：** 引擎不配置 LLM 时，所有功能正常运行（退化为纯算法/规则）
- **无状态调用：** 引擎内部的 LLM 调用不维护上下文，每次独立
- **结果缓存：** 相同输入的 LLM 结果缓存在数据引擎的 DuckDB 中，避免重复调用
- **有状态对话仅在专家层：** Memory 和对话历史只在专家层管理

### Memory 归属

| 组件 | Memory 类型 | 存储位置 |
|------|------------|---------|
| 业务引擎 | LLM 调用缓存（KV） | 数据引擎 DuckDB `llm_cache` 表 |
| 专家层 | 对话历史（Conversation） | 数据引擎 DuckDB `chat_history` 表 |
| 专家层 | 长期记忆（RAG，远期） | 向量数据库（独立，远期） |

## 核心需求：板块资金轮动追踪

### 用户场景

真实投资路径：电网 → 电力 → 电池 → 绿电。投资者持有电网股时，市场热点依次转向电力、电池、绿色能源。**核心痛点：无法提前感知资金流向的下一个方向。**

期望能力：
- 观察多日板块资金流入/流出趋势
- 识别正在形成的热点板块
- 预判资金轮动的下一站（如：电力+绿色 → 绿电成热点）

### 可视化策略

**全市场 3D 地形可视化的局限性（已验证）：**
- 5500+ 只股票的 768 维 BGE 嵌入经 UMAP 降至 2D，信息损失严重
- 实测：余弦相似度排名 550/5513 的两只股票在 2D 空间中被映射为邻居（UMAP 伪影）
- 全市场 3D 可视化无法可靠反映股票间真实关系，**不适合作为投资决策工具**

**保留聚类引擎的价值：**
- 聚类本身（高维空间）仍有分析价值 — 用于认知学习和行业理解
- 板块级别的聚合分析（资金流入/流出、涨跌统计）不依赖降维可视化
- 3D 地形作为探索性工具保留，但不是核心交互界面

**前端方向调整：**
- 弱化全市场 3D 地形的决策权重
- 强化板块级别的资金流向面板（热力图、桑基图、时间轴）
- 聚类结果用于板块划分，而非空间映射

### 多引擎如何服务资金轮动

| 引擎 | 对资金轮动的贡献 |
|------|-----------------|
| 聚类引擎 | 板块划分（哪些股票属于同一板块）、子行业细分 |
| 量化引擎 | 板块级资金流入/流出量化指标、动量信号 |
| 信息引擎 | 政策/新闻驱动的板块催化剂识别 |
| 专家层 | 综合三面分析，给出板块轮动预判 |

### 多引擎作为对冲策略

资金轮动的时序预测本身具有高度不确定性。多引擎架构的意义：
- **不把所有赌注押在一个分析维度上** — 基本面/消息面/技术面各有盲区
- 任何单一引擎失效时，其他引擎仍提供价值
- 引擎间的交叉验证比单一维度更可靠

## 设计约束

- 各引擎通过 Python 接口通信（同进程），未来可拆为微服务
- 每个引擎有自己的 schemas、routes、核心逻辑
- 数据引擎是共享基础设施，其他引擎都依赖它
- MCP Server 是各引擎对外的 AI 接口，保持 stdio transport
- 新增引擎不破坏现有 API 路径
- **LLM 是可选能力，不是硬依赖** — 任何引擎在无 LLM 配置时必须能正常工作
- **全市场可视化是探索工具，不是决策工具** — 高维降维必然损失信息
